## Goal
Reduce tree serialization overhead from 5-11x to within 3x of libgit2 across all languages.

## Benchmark Data
| Operation | libgit2 | Rust | Swift | Kotlin | Worst gap |
|-----------|---------|------|-------|--------|-----------|
| tree_serialize_1k | 0.088ms | 1.005ms | 0.548ms | 0.442ms | Rust 11x |
| tree_serialize_10k | 0.929ms | 10.706ms | 5.425ms | 3.460ms | Rust 12x |

## Root Cause
libgit2 builds tree entries directly into a raw byte buffer with minimal overhead. muongit implementations use higher-level string formatting (`format!`, `String(format:)`, `.format()`) and dynamic buffer growth. Kotlin is closest (5x) likely due to JIT optimization.

## Acceptance Criteria
- [ ] Tree serialize 1K within 3x of libgit2 for all languages
- [ ] All tree serialization/parsing tests pass
- [ ] No API changes

## Implementation Plan
- [ ] Rust: pre-allocate output buffer, use `write!` to a Vec<u8> instead of format!, avoid per-entry String allocations
- [ ] Swift: use ContiguousArray or Data with pre-allocated capacity, write mode/name/oid bytes directly
- [ ] Kotlin: pre-size ByteArrayOutputStream, minimize string allocations in entry formatting
- [ ] Re-run benchmarks and update report

## Notes
- Interesting that Kotlin (3.7x) beats Swift (6x) and Rust (12x) — JIT string formatting is well-optimized
- The benchmark includes both serialize + parse (hash), so SHA overhead is a factor here too

## Progress Log
- 2026-03-18: Tree serialization optimized across all languages. Results (tree_serialize_10k): Rust 0.64ms (was 10.7ms, now faster than libgit2), Kotlin 2.6ms (2.6x), Swift 3.5ms (3.6x). Key fixes: (1) Rust: from_bytes was doing raw→hex→parse hex→raw round-trip, fixed to direct construction. (2) All: pre-allocated buffers, pre-computed mode byte strings, allocation-free sort comparators. (3) Swift: zero-copy tree parsing with UnsafeRawBufferPointer, inline octal parse. (4) Swift: pointer-based OID init to avoid intermediate Array copy.
