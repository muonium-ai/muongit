## Goal
Reduce commit serialization from 5-43x slower to within 3x of libgit2, with focus on Swift (43x worst).

## Benchmark Data
| Operation | libgit2 | Rust | Swift | Kotlin |
|-----------|---------|------|-------|--------|
| commit_serialize_10k | 3.421ms | 16.207ms (4.7x) | 148.013ms (43x) | 48.158ms (14x) |

## Root Cause
- **Swift (43x)**: String interpolation with `\(variable)` is extremely slow for repeated use. Each interpolation creates temporary String objects and involves Unicode validation.
- **Kotlin (14x)**: String templates `"$var"` have similar overhead at scale.
- **Rust (4.7x)**: `format!` macro has moderate overhead from formatting traits.
- **libgit2**: Uses `snprintf` directly into a pre-allocated buffer — minimal allocation.

## Acceptance Criteria
- [ ] Swift commit serialize within 5x of libgit2 (from 43x)
- [ ] Kotlin commit serialize within 5x of libgit2 (from 14x)
- [ ] Rust commit serialize within 3x of libgit2 (from 4.7x)
- [ ] All commit tests pass

## Implementation Plan
- [ ] Swift: replace string interpolation with direct byte buffer writes using `append(contentsOf:)` on Data/[UInt8]
- [ ] Kotlin: replace string templates with StringBuilder or direct ByteArray writes
- [ ] Rust: use `write!` into a pre-allocated Vec<u8> instead of `format!` + concatenation
- [ ] Profile to identify whether the bottleneck is formatting, allocation, or both
- [ ] Re-run benchmarks and update report

## Notes
- Swift is the critical fix — 43x is the largest non-SHA gap
- The benchmark serializes 10K commits per iteration, so per-commit overhead is amplified
