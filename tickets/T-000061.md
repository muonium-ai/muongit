## Goal
Produce a repo-local gap matrix that compares how close each muongit implementation is to stock Git for the workflows users actually care about.

## Acceptance Criteria
- [x] A documented matrix exists in the repo that compares stock Git, the Rust `muongit` CLI, the Swift port, and the Kotlin port across these workflows: clone, branch, everyday commit flow, conflict handling, history inspection, and advanced repo surgery.
- [x] Each workflow row distinguishes end-user usability from underlying engine/library support, so the matrix makes the Rust CLI vs Swift/Kotlin library-only split explicit instead of implying equal user readiness.
- [x] Each matrix cell is grounded in current code/docs/tests and classifies the state with a concrete label such as `usable`, `partial`, `library-only`, or `missing`, plus short notes on the main blockers or limits.
- [x] The output includes a short summary of the highest-priority product gaps that keep muongit from feeling like Git in normal user workflows, with suggested follow-up ticket areas.

## Notes
- This is a product-readiness artifact, not a marketing summary. It should be honest about what a developer can do today without custom glue.
- Treat “advanced repo surgery” as the set of workflows power users reach for after the happy path: merge/rebase/cherry-pick/revert/stash/worktree-style operations, repo recovery, and other command surfaces that make Git feel complete.
- The matrix should prefer what users can do directly today over what the underlying port could theoretically support with additional wrappers.
- Keep the output compact and scannable. One strong matrix plus a short findings section is better than a long prose essay.
