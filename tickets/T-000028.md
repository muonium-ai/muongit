## Goal
Replace pure software SHA-1/SHA-256 with platform-accelerated crypto to close the 108-1364x gap vs libgit2.

## Benchmark Data
| Operation | libgit2 | Rust | Swift | Kotlin | Worst gap |
|-----------|---------|------|-------|--------|-----------|
| sha1_10kb | 0.028ms | 0.098ms | 0.047ms | 3.338ms | Kotlin 118x |
| sha1_1mb | 1.032ms | 111ms | 113ms | - | Rust 108x |
| sha1_10mb | 9.784ms | 13,344ms | 13,511ms | - | Swift 1381x |

## Root Cause
All three muongit ports use pure software SHA-1/SHA-256 implementations. libgit2 delegates to OpenSSL or platform crypto (CommonCrypto on macOS) which uses hardware-accelerated SHA instructions (ARM SHA extensions on Apple Silicon).

## Acceptance Criteria
- [ ] Rust: use `ring` or `sha1`/`sha2` crates with hardware acceleration
- [ ] Swift: use CommonCrypto CC_SHA1/CC_SHA256 or CryptoKit
- [ ] Kotlin: use `java.security.MessageDigest` (JVM has native SHA)
- [ ] All existing SHA tests pass with new implementations
- [ ] Benchmark SHA-1 1MB within 2x of libgit2 for Rust and Swift

## Implementation Plan
- [ ] Rust: add `sha1` and `sha2` crates as dependencies, replace SHA1/SHA256 internals
- [ ] Swift: import CommonCrypto, replace SHA1.hash/SHA256Hash.hash with CC_SHA1/CC_SHA256
- [ ] Kotlin: replace pure SHA with java.security.MessageDigest("SHA-1") / ("SHA-256")
- [ ] Keep pure implementations as fallback (feature-gated in Rust, conditional in others)
- [ ] Re-run benchmarks and update report

## Notes
- This is the highest-impact optimization — it will also fix OID creation, blob hashing, and index R/W performance
- Kotlin has no 1MB/10MB SHA benchmarks because pure impl is ~37s/MB
- libgit2 uses OpenSSL on Linux, CommonCrypto/SecureTransport on macOS
