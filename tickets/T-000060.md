## Goal
Create a repeatable end-to-end repository workflow that each implementation can author and every other implementation can read and validate accurately across the full feature surface.

## Acceptance Criteria
- [x] A deterministic conformance fixture or scenario runner creates a brand-new repository with each writer implementation in scope, then validates that every other reader implementation can open and inspect that repository without loss of meaning.
- [x] The workflow exercises the shipped feature surface, including object reads, staging/add-remove/commit, branch/refdb operations, revision parsing and history walking, switch/reset/restore flows, patch generation/apply behavior, and remote clone-fetch-push round trips.
- [x] Validation is semantic, not just file-exists smoke coverage: the reader checks must assert commit graph shape, HEAD and ref state, object IDs, tree contents, status/index state, revparse results, generated patches, and remote synchronization outcomes.
- [x] The test matrix is repeatable inside the repo sandbox using `tmp/`, with deterministic fixtures/artifacts and a documented command that runs the full cross-implementation workflow.
- [x] Failures identify the writer step and the reader implementation that disagreed, so parity regressions are actionable instead of opaque.

## Notes
- Treat this as a product-confidence ticket, not a unit-test expansion. The point is to prove repository interoperability end-to-end across implementations, not just per-port local correctness.
- The strongest shape is a writer/reader matrix:
- Writers: at minimum `libgit2`, `rust`, `swift`, and `kotlin` if practical.
- Readers: every non-writing implementation must open the repo and verify the expected semantic state after each authored phase.
- The authored workflow should cover the feature areas added in the recent parity wave:
- Init/open and object API reads.
- Add/remove/commit porcelain and status/index transitions.
- Branch/refdb and detached-HEAD behavior.
- Revision parsing and revwalk/history assertions.
- Switch/checkout/reset/restore flows.
- Patch formatting/apply flows.
- Remote transport clone/fetch/push interoperability.
- If one giant scenario becomes unmanageable, split it into a small set of canonical phases with persisted checkpoints, but keep them part of one blessed conformance workflow.
